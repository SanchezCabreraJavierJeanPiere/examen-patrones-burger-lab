Singleton

Creacional · GoF

Garantizar que una clase tenga UNA sola instancia y dar un punto de acceso global a ella.

📍 Dónde está

ArchivoRol
proyecto_final/SistemaHamburgueseria.javaLa clase Singleton (líneas 11–24: holder + constructor privado + getInstancia)
proyecto_final/MainSistemaHamburgueseria.javaDemostración: sistema1 == sistema2 imprime true (líneas 11–17)

🔑 El código que lo hace ser Singleton

SistemaHamburgueseria.java (líneas 11–24)// Holder idiom: la instancia se crea al solicitarla por primera vez.
private static class ContenedorInstancia {
    private static final SistemaHamburgueseria INSTANCIA = new SistemaHamburgueseria();
}

// Constructor privado: otras clases no pueden crear mas sistemas.
private SistemaHamburgueseria() {
    inventario = new Inventario();
}

public static SistemaHamburgueseria getInstancia() {
    // Siempre devuelve exactamente el mismo objeto.
    return ContenedorInstancia.INSTANCIA;
}
Son 3 piezas juntas las que forman el patrón: ① constructor private (nadie más puede hacer new), ② campo estático con la única instancia, ③ método público estático getInstancia() como único punto de acceso. Si falta cualquiera de las tres, ya no es Singleton.

🎯 Qué problema resuelve aquí

El Inventario (stock de pan, carne, queso…) debe ser uno solo para toda la aplicación. Si cada pantalla o cada pedido creara su propio SistemaHamburgueseria, cada uno tendría su propio inventario y los descuentos de stock no se verían entre sí: venderíamos hamburguesas con stock fantasma.

⚔️ Por qué así y no de otra forma

¿Por qué "holder idiom" y no el Singleton clásico con if?
El clásico hace if (instancia == null) instancia = new ... — ese if no es thread-safe (dos hilos pueden entrar a la vez y crear dos instancias) y además el proyecto evita condicionales. El holder usa una clase interna estática: la JVM garantiza que una clase se inicializa una sola vez y de forma segura entre hilos, recién cuando alguien la usa por primera vez. Resultado: lazy + thread-safe + sin synchronized + sin if.
¿Por qué no una clase con métodos static y ya?
Una clase estática no puede implementar interfaces, no se puede pasar como parámetro, no se puede heredar y no permite inicialización perezosa controlada. El Singleton sigue siendo un objeto: aquí se pasa a otros métodos y mantiene estado (inventario).
¿Por qué no eager (crear la instancia al cargar la clase)?
Funcionaría, pero el holder ya da lo mismo y mejor: la instancia se crea solo si alguien la pide. Con eager, cargas el inventario aunque nunca se use.

🎓 Preguntas rebuscadas

¿Qué pasa si dos hilos llaman a getInstancia() exactamente al mismo tiempo?
Nada malo: la JVM garantiza que la inicialización de la clase ContenedorInstancia ocurre una sola vez (el class loader se sincroniza internamente). Ambos hilos reciben la misma INSTANCIA. Por eso no necesitamos synchronized ni double-checked locking.
¿Por qué la clase interna es static? ¿Qué pasaría si no lo fuera?
Si no fuera static, necesitaría una instancia de SistemaHamburgueseria para existir — círculo imposible, no compilaría en este uso. Además, siendo estática, solo se carga cuando getInstancia() la toca por primera vez: eso es lo que hace lazy al patrón.
Señale la línea exacta que impide crear dos sistemas.
Línea 17: private SistemaHamburgueseria(). Al ser privado el constructor, new SistemaHamburgueseria() desde otra clase no compila.
¿Cómo demuestran en código que es una sola instancia?
En MainSistemaHamburgueseria líneas 11–17: se piden dos referencias y se compara con == (identidad, no equals): sistema1 == sistema2 imprime true → son el mismo objeto en memoria.
¿Qué crítica se le hace al Singleton y cómo la defienden?
Se le critica ser un "global encubierto" que dificulta los tests. Defensa: aquí el alcance es pequeño y la alternativa (pasar el inventario por todos los constructores de la UI) complicaría más; además el acceso queda centralizado y controlado en una sola clase coordinadora, no regado por el código.
Respuesta de 20 segundos: "SistemaHamburgueseria es Singleton con holder idiom: constructor privado, clase interna estática con la única instancia y getInstancia() como acceso. Lo usamos porque el inventario debe ser uno solo para toda la app, y el holder nos da lazy y thread-safe sin synchronized y sin if."

Factory Method

Creacional · GoF

Definir una interfaz para crear objetos, dejando que cada fábrica concreta decida qué clase instanciar.

📍 Dónde está

ArchivoRol en el patrón
proyecto_final/FactoryAlimento.javaCreator (interfaz): declara crearAlimento(Inventario)
proyecto_final/FactoryHamburguesa.java
FactoryPapasFritas.java · FactoryHelado.java · FactoryRefresco.java		Concrete Creators: cada una instancia SU producto
proyecto_final/Alimento.javaProduct (interfaz que devuelve la fábrica)
Hamburguesa/PapasFritas/Helado/Refresco.javaConcrete Products
src/patrones/FactoryFigura.java y demopatrones/templatefactory/Demos del mismo patrón con figuras

🔑 El código que lo hace ser Factory Method

FactoryAlimento.javapublic interface FactoryAlimento {
    // Devuelve Alimento para que el cliente no dependa de una clase concreta.
    public Alimento crearAlimento(Inventario inventario);
}
FactoryHamburguesa.java (líneas 6–11)@Override
public Alimento crearAlimento(Inventario inventario) {
    // El cliente recibe la interfaz Alimento, no necesita conocer mas detalles.
    Hamburguesa h = new Hamburguesa(inventario);
    return h;
}
El método fábrica es crearAlimento(). La clave está en la firma: recibe el inventario, devuelve la interfaz Alimento — pero por dentro decide la clase concreta. El único new Hamburguesa(...) de todo el sistema vive aquí.

🎯 Qué problema resuelve aquí

La UI y el main pueden pedir "un alimento" sin saber qué clase concreta se construye ni cómo. Para agregar un producto nuevo (ej. Ensalada) solo se crean 2 clases nuevas (Ensalada + FactoryEnsalada) y no se toca ni una línea del código existente → principio Open/Closed.

⚔️ Por qué así y no de otra forma

¿Por qué no Abstract Factory?
Abstract Factory crea familias de productos relacionados (ej. "combo italiano" = pizza + pasta + vino, vs "combo peruano"). Aquí cada fábrica crea UN solo producto independiente — es exactamente Factory Method. Usar Abstract Factory sería sobre-ingeniería: no hay familias.
¿Por qué no una fábrica única con switch(tipo)?
Un switch("hamburguesa") → new Hamburguesa() funciona, pero cada producto nuevo obliga a modificar ese switch (viola Open/Closed) y concentra todas las dependencias en una clase. Con una fábrica por producto, la decisión la toma el polimorfismo: cero condicionales, cada botón de la UI instancia directamente su fábrica.
¿Por qué no Builder?
Builder es para construcción compleja paso a paso del MISMO objeto (una hamburguesa armada ingrediente por ingrediente con combinaciones opcionales). Aquí los productos se crean de un solo golpe con datos fijos: no hay pasos opcionales que justifiquen un Builder.
¿Por qué no Prototype?
Prototype clona instancias existentes cuando crear desde cero es caro. Crear una Hamburguesa es barato y cada una necesita el inventario actual — clonar no aporta nada.

🎓 Preguntas rebuscadas

¿Por qué crearAlimento() devuelve Alimento y no Hamburguesa?
Programar contra la interfaz, no la implementación (principio de inversión de dependencias). Si devolviera Hamburguesa, el cliente quedaría acoplado a la clase concreta y el patrón perdería su gracia: no podrías tratar todos los productos igual ni agregarlos al pedido de forma uniforme.
¿Por qué la fábrica recibe el Inventario como parámetro?
Es inyección de dependencia: el producto necesita el inventario para descontar ingredientes en comprobarIngredientes(). La fábrica se lo entrega en el constructor (new Hamburguesa(inventario)), así el producto nace listo y no tiene que buscar el inventario por su cuenta.
Entonces… ¿dónde se decide QUÉ fábrica usar? ¿No hay un if escondido?
La decisión la toma el usuario al hacer clic: cada botón de la interfaz (o cada línea del main) instancia directamente su fábrica concreta (new FactoryHamburguesa()). No hay if/switch: la selección física del botón reemplaza al condicional.
¿Cuántos 'new Hamburguesa' hay en el sistema y por qué importa?
Uno solo, dentro de FactoryHamburguesa.crearAlimento(). Importa porque el punto de creación está centralizado: si mañana la hamburguesa necesita otro parámetro, se cambia en UN lugar.
¿Qué principio SOLID demuestra este patrón aquí?
Open/Closed: abierto a extensión (nuevas fábricas y productos), cerrado a modificación (el código cliente no se toca). También DIP: el cliente depende de las abstracciones FactoryAlimento y Alimento, no de clases concretas.
Respuesta de 20 segundos: "FactoryAlimento es la interfaz con el método fábrica crearAlimento(); hay una fábrica concreta por producto. El cliente recibe la interfaz Alimento y nunca hace new de un producto. Así agregamos productos sin modificar código existente y sin condicionales."

Template Method

Comportamiento · GoF

Definir en una clase base el esqueleto de un algoritmo, dejando pasos específicos a las subclases sin que puedan cambiar el orden.

📍 Dónde está

ArchivoRol en el patrón
proyecto_final/AlimentoAbstracto.javaClase abstracta: el método plantilla preparar() (líneas 21–28)
Hamburguesa/PapasFritas/Helado/Refresco.javaSubclases: implementan los pasos abstractos
src/demopatrones/template/FiguraAbstracta.javaDemo del mismo patrón con figuras (color, trazo, pintar)

🔑 El código que lo hace ser Template Method

AlimentoAbstracto.java (líneas 21–46)// Metodo plantilla: el orden de estos pasos es igual para todos los productos.
public void preparar() {
    System.out.println("Preparando: " + nombre);
    comprobarIngredientes();   // ← paso abstracto (varía)
    prepararIngredientes();    // ← paso abstracto (varía)
    cocinar();                 // ← paso abstracto (varía)
    empacar();                 // ← paso común (fijo)
    entregar();                // ← paso común (fijo)
}

public abstract void comprobarIngredientes();
public abstract void prepararIngredientes();
public abstract void cocinar();

public void empacar()  { System.out.println("Se empaca " + nombre); }
public void entregar() { System.out.println("Se entrega " + nombre); }
El patrón ES el método preparar(): fija el orden invariable (comprobar → preparar → cocinar → empacar → entregar) y mezcla pasos abstract (cada producto los rellena) con pasos concretos reutilizados. La Hamburguesa cocina carne; el Helado "no necesita cocinarse" — pero nadie puede saltarse el orden.

🎯 Qué problema resuelve aquí

Los 4 productos se preparan con la misma secuencia pero con detalles distintos. Sin el patrón, cada producto repetiría el flujo completo (código duplicado) o un método gigante tendría if por tipo de producto. Con el patrón: el flujo se escribe UNA vez y cada producto solo aporta sus diferencias.

⚔️ Por qué así y no de otra forma

¿Por qué no Strategy? (LA pregunta clásica)
Ambos varían comportamiento, pero: Template Method usa herencia y varía PASOS dentro de un orden fijo definido por la superclase; Strategy usa composición y reemplaza el algoritmo COMPLETO, pudiendo cambiarlo en tiempo de ejecución. Aquí el orden de preparación jamás cambia y un producto no cambia su forma de prepararse a mitad de la vida — herencia con esqueleto fijo es lo natural. (En cambio, el pago SÍ se cambia en runtime: por eso ese lado usa composición.)
¿Por qué no un método con if por producto?
if (tipo == "helado") no cocinar... concentra todos los casos en un solo método que crece con cada producto y viola Open/Closed. Con Template, agregar un producto = una subclase nueva; el esqueleto no se toca.

🎓 Preguntas rebuscadas

Hay DOS métodos preparar() en el proyecto. ¿Son el mismo patrón?
No, y esta es la trampa favorita. ① AlimentoAbstracto.preparar() es el Template Method (esqueleto con pasos). ② Pedido.preparar() viene de la interfaz Alimento y es la operación del Composite: recorre la lista de alimentos hijos y llama al preparar() de cada uno. El mismo nombre, dos patrones distintos: uno define CÓMO se prepara un producto; el otro PROPAGA la orden a todos los hijos.
¿Qué pasos son abstractos y cuáles concretos? ¿Por qué esa división?
Abstractos: comprobarIngredientes(), prepararIngredientes(), cocinar() — es lo que cambia por producto. Concretos: empacar() y entregar() — son iguales para todos y se reutilizan sin repetir código. La división refleja exactamente qué varía y qué no.
¿Qué defecto tiene su preparar() según el patrón "de libro"?
Debería ser final: tal como está, una subclase podría sobrescribir preparar() y romper el orden del esqueleto (que es justo lo que el patrón promete proteger). Respuesta honesta: "Es una mejora que identificamos: agregar final a la firma. Ninguna subclase lo sobrescribe, así que el comportamiento actual es correcto."
El Refresco no se cocina. ¿No rompe eso el patrón?
No: Refresco.cocinar() se implementa imprimiendo "no necesita cocinarse" — el paso existe y se ejecuta en su orden, solo que su contenido es trivial. En la teoría esto se acerca a un hook: un paso que las subclases pueden llenar o dejar casi vacío.
¿Dónde más aparece Template Method en el proyecto?
En demopatrones/template/FiguraAbstracta.java (demo de clase): pasos abstractos obtenerColor(), obtenerTrazo(), pintarFigura() con figuras como Círculo, Rectángulo y Hexágono. Y demopatrones/templatefactory/ lo combina con Factory Method.
Respuesta de 20 segundos: "AlimentoAbstracto.preparar() es el método plantilla: fija el orden comprobar→preparar→cocinar→empacar→entregar. Los tres primeros son abstractos y cada producto los implementa; los dos últimos son comunes. Elegimos Template y no Strategy porque el orden es fijo y varían pasos, no el algoritmo completo."

Composite

Estructural · GoF

Componer objetos en estructuras árbol y tratar igual a un objeto individual que a un grupo completo.

📍 Dónde está

ArchivoRol en el patrón
proyecto_final/Alimento.javaComponent: interfaz común (preparar(), getNombre(), getPrecio())
proyecto_final/Pedido.javaComposite: tiene List<Alimento> y también ES un Alimento
Hamburguesa/PapasFritas/Helado/Refresco.javaLeaf (hojas): productos individuales
src/patrones/Compuesto.javaDemo: un Compuesto pinta una lista de Figura

🔑 El código que lo hace ser Composite

Pedido.java (líneas 10, 36–41, 61–77)public abstract class Pedido extends PedidoAbstracto implements Alimento {

    private List<Alimento> alimentos;   // hijos guardados por la INTERFAZ

    public void agregarAlimento(Alimento alimento) {
        alimentos.add(alimento);
        monto += alimento.getPrecio();
    }

    @Override
    public void preparar() {
        cambiarEstado(EN_PREPARACION);
        try {
            // Composite: la misma operacion se aplica a todos los hijos.
            for (Alimento alimento : alimentos) {
                alimento.preparar();
            }
            cambiarEstado(LISTO);
        } catch (StockInsuficienteException e) {
            cambiarEstado(SIN_STOCK);
        }
    }
}
Dos señales delatan al Composite: ① Pedido implementa la misma interfaz que sus hijos (implements Alimento) — un pedido "es un" alimento; ② preparar() del pedido propaga la operación a cada hijo con un for. El cliente llama pedido.preparar() igual que llamaría hamburguesa.preparar(): tratamiento uniforme.

🎯 Qué problema resuelve aquí

Un pedido agrupa varios productos, pero el sistema quiere tratarlo como una unidad: prepararlo, ponerle nombre, cobrarle un precio. Gracias al Composite, SistemaHamburgueseria.coordinarPreparacion() hace UNA llamada y no necesita saber cuántos productos hay dentro ni de qué tipo son.

⚔️ Por qué así y no de otra forma

¿Por qué no Decorator? (se parecen: ambos "envuelven")
Decorator envuelve UN solo objeto para añadirle responsabilidades (ej. HamburguesaConQuesoExtra que envuelve una Hamburguesa y suma 2 soles). Composite agrega MUCHOS hijos para tratarlos como uno. El Pedido no le añade comportamiento a una hamburguesa: la agrupa con otras. La lista List<Alimento> (muchos) vs una referencia única (uno) es la diferencia estructural visible.
¿Por qué no una simple lista en el cliente (sin patrón)?
Si la UI manejara una List<Alimento> a mano, cada operación (preparar, calcular total, mostrar) repetiría el bucle for en cada lugar donde se use. El Composite encapsula el recorrido dentro del propio Pedido y además permite que un pedido se use donde se espera un alimento.

🎓 Preguntas rebuscadas

¿Un Pedido podría contener OTRO Pedido dentro?
Sí — y esa es la prueba de fuego del Composite: como Pedido implements Alimento y agregarAlimento() recibe cualquier Alimento, un pedido puede ser hijo de otro (ej. un pedido corporativo que agrupa los pedidos de cada empleado). La estructura es recursiva (árbol). La UI actual no lo usa, pero el diseño lo permite sin cambiar una línea.
Identifique Component, Composite y Leaf con nombres de clases.
Component: la interfaz Alimento. Composite: Pedido (y sus hijas PedidoMostrador/PedidoDelivery). Leaf: Hamburguesa, PapasFritas, Helado, Refresco.
getPrecio() del Pedido devuelve 'monto'. ¿Eso es Composite puro?
Confesión técnica: el Composite "de libro" haría return alimentos.stream().sum(getPrecio) — recorrer los hijos. Aquí se usa monto, un acumulador que se suma en agregarAlimento(). El resultado es el mismo, pero si alguien quitara un alimento de la lista sin ajustar el monto, se desincronizaría. Respuesta honesta: "Es una optimización con acumulador; la versión más pura sumaría los hijos al momento, y es una mejora identificada."
¿Dónde ejercita el sistema el tratamiento uniforme?
En Pedido.preparar(): el for llama alimento.preparar() sin saber si el hijo es Hamburguesa o Helado — y también en agregarAlimento(), que suma alimento.getPrecio() por la interfaz. El demo patrones/Compuesto.java hace lo mismo con figuras: pintar() recorre y pinta cada una.
Respuesta de 20 segundos: "Alimento es el Component, los productos son las hojas y Pedido es el Composite: implementa Alimento y guarda List<Alimento>. Su preparar() propaga la operación a cada hijo. Así el sistema trata a un pedido completo igual que a un producto individual."

Observer

Comportamiento · GoF

Definir una dependencia uno-a-muchos: cuando un objeto cambia de estado, todos sus suscriptores son notificados automáticamente.

📍 Dónde está

ArchivoRol en el patrón
proyecto_final/ObservadorPedido.javaObserver (interfaz): declara notificar(Pedido)
proyecto_final/Pedido.javaSubject/Publicador: lista de observadores + suscribir() + notificarObservadores() (líneas 43–59)
Cocina.java · Despacho.java · PantallaCliente.javaObservers concretos
SistemaHamburgueseria.java (líneas 36–44)Aquí se SUSCRIBEN los observadores al crear el pedido
src/patrones/observer/Demo electoral: Publicador, ObservadorElectoral, DataElectoral

🔑 El código que lo hace ser Observer

Pedido.java (líneas 43–59)public void suscribir(ObservadorPedido observador) {
    observadores.add(observador);
}

public void notificarObservadores() {
    // Todos reciben el mismo pedido sin conocer sus implementaciones concretas.
    for (ObservadorPedido observador : observadores) {
        observador.notificar(this);
    }
}

public void cambiarEstado(String estado) {
    this.estado = estado;
    notificarObservadores();   // ← cada cambio se comunica automáticamente
}
SistemaHamburgueseria.java (líneas 36–41) — la suscripciónprivate Pedido configurarPedido(Pedido pedido, String tipoPedido) {
    pedido.suscribir(new Cocina());
    pedido.suscribir(new Despacho());
    pedido.suscribir(new PantallaCliente());
    pedido.cambiarEstado(Pedido.REGISTRADO);
    return pedido;
}
El disparador es cambiarEstado(): cada cambio de estado (REGISTRADO → EN_PREPARACION → LISTO/SIN_STOCK → ENTREGADO) llama a notificarObservadores(), que recorre la lista y avisa. El Pedido conoce a sus observadores solo por la interfaz: no sabe (ni le importa) si son cocinas o pantallas.

🎯 Qué problema resuelve aquí

Cocina, Despacho y la Pantalla del cliente necesitan enterarse del avance del pedido. Sin Observer, el Pedido tendría que llamar explícitamente a cada área (cocina.avisar(); despacho.avisar();…) quedando acoplado a todas; agregar un área nueva (ej. Contabilidad) obligaría a modificar Pedido. Con Observer: se crea la clase, se suscribe, y listo.

⚔️ Por qué así y no de otra forma

¿Por qué no que el sistema pregunte el estado cada cierto tiempo (polling)?
Con polling, cada área revisaría el pedido en bucle: procesamiento inútil y avisos tardíos. Con Observer, el aviso es inmediato y solo cuando algo cambió — el flujo de control se invierte: "no me llames, yo te llamo".
¿Por qué no Mediator?
Mediator centraliza comunicación bidireccional entre muchos colegas que se hablan entre sí. Aquí la comunicación es uno-a-muchos y en un solo sentido: el pedido emite, los demás escuchan; Cocina jamás le habla a Despacho. Observer es el ajuste exacto.
¿Por qué no eventos de JavaFX directamente?
Los observadores son objetos del dominio (Cocina, Despacho), no widgets. Implementar el patrón a mano lo mantiene independiente de la tecnología de interfaz: funciona igual en el main de consola que en la app JavaFX.

🎓 Preguntas rebuscadas

¿Quién es el Subject y quién el Observer? Nómbrelos con precisión.
Subject (publicador): Pedido — mantiene la lista, expone suscribir() y dispara notificarObservadores(). Observer (interfaz): ObservadorPedido con su único método notificar(Pedido). Observers concretos: Cocina, Despacho, PantallaCliente.
¿En qué línea exacta se dispara la notificación?
Pedido.java línea 58: dentro de cambiarEstado() se llama notificarObservadores(). Por eso ningún cambio de estado puede pasar desapercibido: el aviso está incrustado en el único método que cambia el estado.
¿Su Observer es modelo push o pull?
Híbrido con espíritu pull: se envía el objeto completo (notificar(this)) y cada observador extrae lo que le interesa (pedido.getEstado(), getCliente()…). Un push puro enviaría solo los datos exactos (ej. el string del estado). Ventaja del enfoque: si mañana un observador necesita otro dato del pedido, no cambia la interfaz.
¿Qué le falta a su implementación respecto al patrón completo?
El método desuscribir() (detach). GoF define attach/detach; aquí solo hay attach porque los observadores viven lo mismo que el pedido. Respuesta honesta: "Lo omitimos porque nuestros observadores nunca se dan de baja; agregarlo es trivial: observadores.remove(obs)."
¿Dónde se suscriben los observadores y por qué ahí?
En SistemaHamburgueseria.configurarPedido() (líneas 36–44), inmediatamente al registrar el pedido y ANTES del primer cambiarEstado(REGISTRADO) — así ningún observador se pierde ni siquiera el primer estado. Lo cablea SistemaHamburgueseria, para que ni la UI ni el Pedido conozcan ese detalle.
Respuesta de 20 segundos: "Pedido es el subject: guarda ObservadorPedido en una lista, y cambiarEstado() dispara notificar(this) a todos. Cocina, Despacho y PantallaCliente son los observadores, suscritos en SistemaHamburgueseria al registrar el pedido. Así el pedido avisa sin acoplarse a ningún área concreta."

Bridge

Estructural · GoF

Desacoplar una abstracción de su implementación para que ambas jerarquías evolucionen de forma independiente.

🔑 El código que lo hace ser Bridge

PedidoAbstracto.java (líneas 7–31, resumido)public abstract class PedidoAbstracto {

    protected MetodoPago metodoPago;   // ← EL PUENTE: composición, no herencia

    public abstract void procesarPago();

    public void setMetodoPago(MetodoPago metodoPago) {
        // Permite cambiar la implementacion de pago en tiempo de ejecucion.
        this.metodoPago = metodoPago;
    }
}
PedidoDelivery.java (líneas 26–35)@Override
public void procesarPago() {
    // Bridge: el tipo de pedido calcula el total y MetodoPago realiza el cobro.
    metodoPago.pagar(getPrecio());
}

@Override
public double getPrecio() {
    return super.getPrecio() + costoEnvio;   // delivery agrega su envío
}
El puente es el campo protected MetodoPago metodoPago. Hay dos jerarquías independientes: la de pedidos (Mostrador, Delivery) que decide QUÉ monto cobrar, y la de pagos (Tarjeta, Yape, Efectivo) que decide CÓMO cobrarlo. La línea del patrón: metodoPago.pagar(getPrecio()) — la abstracción delega en la implementación.

🎯 Qué problema resuelve aquí

Sin Bridge, cada combinación necesitaría una clase: PedidoMostradorTarjeta, PedidoDeliveryYape… 2 tipos × 3 pagos = 6 clases (y crecería multiplicando). Con Bridge: 2 + 3 = 5 clases y cualquier combinación se arma en runtime con setMetodoPago(). Agregar PagoPlin = 1 clase nueva, cero modificaciones.

⚔️ Por qué así y no de otra forma — incluye LA pregunta asesina

🔥 "Esto es Strategy, no Bridge" — ¿cómo lo defiendo?
Estructuralmente son casi idénticos (interfaz + composición + delegación), la diferencia es de intención y de jerarquías:

· Strategy: UN contexto intercambia algoritmos equivalentes. El foco es variar un comportamiento.
· Bridge: DOS jerarquías completas diseñadas para crecer por separado. El foco es la estructura: evitar la explosión combinatoria de clases.

Aquí hay jerarquía en AMBOS lados: pedidos (Mostrador/Delivery, cada uno con su cálculo de precio) y pagos (Tarjeta/Yape/Efectivo). Ambas crecen independientes → Bridge. Si solo existiera la clase Pedido con pagos intercambiables, sería Strategy. Remate honesto: "Bridge y Strategy comparten estructura; los distingue la intención — y la nuestra es desacoplar dos jerarquías que evolucionan solas. De hecho GoF lo dice: muchos patrones difieren solo en su propósito."
¿Por qué no herencia pura (subclases por combinación)?
Explosión de clases: cada tipo de pedido nuevo multiplica por todos los pagos y viceversa (n×m). Además la combinación quedaría fija en compilación; con el puente, el cliente elige el pago con un clic en runtime (setMetodoPago(new PagoYape(...))).
¿Por qué no un switch(tipoPago) dentro de Pedido?
Cada nuevo método de pago obligaría a modificar Pedido (viola Open/Closed) y Pedido acumularía la lógica de todos los pagos (viola responsabilidad única). Con Bridge, cada pago vive en su propia clase.

Señale la línea exacta donde "cruza el puente".
PedidoMostrador.java línea 20 (y PedidoDelivery línea 29): metodoPago.pagar(getPrecio()). Ahí la abstracción (pedido) delega el trabajo a la implementación (pago) a través de la interfaz.
¿Qué aporta cada lado del puente? Demuestre que ambos varían.
Lado abstracción: PedidoDelivery.getPrecio() suma costoEnvio; PedidoMostrador no. Lado implementación: PagoTarjeta imprime el número de tarjeta, PagoYape el celular, PagoEfectivo solo el monto. Dos ejes de variación reales e independientes: la definición de Bridge.

En PedidoAbstracto.setMetodoPago() (líneas 28–31). En el main, línea 42: pedido.setMetodoPago(new PagoTarjeta("123456789")). En la app JavaFX, al elegir el método de pago en pantalla.
¿Por qué PedidoAbstracto es clase abstracta y MetodoPago es interfaz?
La abstracción comparte estado y código entre sus hijas (el campo metodoPago, monto, los setters) → clase abstracta. El implementor solo define un contrato de comportamiento sin estado común (pagar(monto)) → interfaz. Es la forma canónica del Bridge en Java.

✅ Buenas prácticas del proyecto

Además de los patrones, el proyecto aplica principios de diseño. El profe puede preguntar por ellos con nombre propio.

⭐ Responsabilidad Única (SRP) — la estrella

Qué significa: es la "S" de SOLID (Robert C. Martin): una clase debe tener UNA sola responsabilidad — una sola razón para cambiar. Si puedes describir la clase sin usar "y" ("crea hamburguesas y cobra y maneja stock"), va bien; si necesitas el "y", la clase hace demasiado.

Dónde se cumple en el proyecto — cada clase se describe en una sola frase sin "y":

ClaseSu ÚNICA responsabilidad
FactoryHamburguesaCrear hamburguesas. Nada más.
PagoTarjeta / PagoYape / PagoEfectivoCada uno sabe cobrar de UNA forma
InventarioGestionar el stock de ingredientes (verificar, descontar, agregar)
Cocina / Despacho / PantallaClienteCada uno reacciona a los estados del pedido desde SU área
PedidoAdministrar su contenido y su ciclo de estados
SistemaHamburgueseriaCoordinar las operaciones del negocio (delega, no ejecuta)
StockInsuficienteExceptionRepresentar un único error del dominio
Por qué se usó: ① cambios localizados — si mañana Yape cambia su API, solo se toca PagoYape; cero riesgo de romper cocina o inventario. ② clases cortas y legibles — ninguna clase del dominio pasa de ~80 líneas. ③ los patrones lo producen naturalmente: Factory Method separa "crear" de "usar", Observer separa "avisar" de "reaccionar", Bridge separa "qué cobrar" de "cómo cobrar". Patrones y SRP se refuerzan mutuamente.

🧰 Las demás buenas prácticas (con evidencia en el código)

Programar contra interfaces, no implementaciones (DIP)
El cliente solo conoce Alimento, MetodoPago, FactoryAlimento y ObservadorPedido. Las clases concretas se instancian en un único punto cada una. Evidencia: crearAlimento() devuelve Alimento, nunca Hamburguesa.
Open/Closed: extender sin modificar
Producto nuevo = 2 clases nuevas (producto + fábrica); método de pago nuevo = 1 clase nueva; área observadora nueva = 1 clase + 1 suscribir(). En ningún caso se edita código existente.
Cero if/switch de tipos: polimorfismo
El dominio no tiene condicionales para decidir tipos de producto, de pago ni de pedido — cada decisión vive en una clase polimórfica. Verificable con una búsqueda de if/switch en proyecto_final.
Encapsulamiento defensivo
Inventario.getIngredientes() (línea 71) devuelve Collections.unmodifiableMap(...): la UI puede LEER el stock pero no modificarlo por fuera; toda mutación pasa por los métodos del inventario. Además descontarIngredientes() (líneas 44–59) valida todo antes de descontar, para nunca dejar un descuento a medias.
Inventario.java (líneas 69–72)public Map<String, Integer> getIngredientes() {
    // Vista de solo lectura: la interfaz consulta, pero no modifica el mapa.
    return Collections.unmodifiableMap(ingredientes);
}
Excepción propia del dominio
StockInsuficienteException le pone nombre de negocio a un error de negocio. Pedido.preparar() la captura y la traduce a un estado (SIN_STOCK) que el Observer difunde — el error no revienta el programa, se convierte en información.
Constantes con nombre en lugar de "strings mágicos"
Los estados viven UNA vez como constantes (Pedido.REGISTRADO, EN_PREPARACION, LISTO, SIN_STOCK, ENTREGADO, líneas 13–17). Un typo en un string suelto compila y falla en silencio; un typo en la constante no compila.
Inyección de dependencias (manual)
Los productos no buscan el inventario: lo reciben por constructor (new Hamburguesa(inventario), entregado por la fábrica). La dependencia es explícita y visible en la firma.
Separación modelo ↔ vista
El mismo modelo sirve a dos vistas sin cambiar una línea: el main de consola (MainSistemaHamburgueseria) y la app JavaFX (interfaz/AplicacionHamburgueseria). Prueba de que la lógica no está mezclada con la interfaz.

🎓 Preguntas rebuscadas

¿Qué significa exactamente "una sola razón para cambiar"?
Que solo UN tipo de requisito puede obligarte a editar esa clase. PagoYape solo cambia si cambia cómo se cobra por Yape; Inventario solo cambia si cambia la gestión de stock. Si una clase cambiara por dos motivos distintos (cobros Y stock), tendría dos responsabilidades mezcladas y cada cambio arriesgaría romper al otro.
Deme un ejemplo de clase que VIOLARÍA SRP en este proyecto.
Un GestorTodo que cree productos, cobre y descuente stock: cambiar la lógica de pago podría romper la creación de productos. El proyecto lo evita repartiendo cada responsabilidad: fábricas crean, pagos cobran, inventario gestiona stock.
¿SRP es lo mismo que "una clase por archivo" o "métodos cortos"?
No. SRP habla de motivos de cambio, no de tamaño. Una clase de 300 líneas con una sola responsabilidad cumple SRP; una de 20 líneas que cobra Y registra stock lo viola. El tamaño pequeño es una consecuencia frecuente, no la definición.
¿Los patrones garantizan SRP automáticamente?
No lo garantizan, pero lo empujan: cada patrón fuerza una separación (crear/usar, avisar/reaccionar, qué/cómo). Aun así se puede violar SRP dentro de un patrón — por ejemplo, si una fábrica además cobrara. La disciplina la pone el diseñador; el patrón solo marca el molde.
¿Qué pasaría si getIngredientes() devolviera el mapa directamente?
Cualquier clase podría hacer getIngredientes().put("Carne", 999) y corromper el stock saltándose la validación de descontarIngredientes(). El unmodifiableMap lanza excepción ante cualquier intento de escritura: la única puerta de mutación son los métodos del propio Inventario.
Respuesta de 20 segundos (SRP): "Aplicamos Responsabilidad Única: cada clase tiene una sola razón para cambiar — cada fábrica solo crea su producto, cada pago solo sabe cobrar de una forma, el inventario solo gestiona stock y los observadores solo reaccionan desde su área. Lo usamos para que un cambio en una parte no arriesgue romper otra, y se nota: ninguna clase del dominio necesita un 'y' para describirse."

🪤 Las trampas favoritas del profe

Preguntas cruzadas que mezclan patrones o atacan los puntos débiles. Piensa la respuesta ANTES de abrir cada una.

"Hay dos preparar() distintos. ¿Cuál es de qué patrón?"
AlimentoAbstracto.preparar() = Template Method (esqueleto de pasos de UN producto). Pedido.preparar() = operación del Composite (viene de la interfaz Alimento, recorre los hijos y llama al preparar de cada uno). El del pedido TERMINA invocando al del producto: los dos patrones cooperan en cadena.
"Clasifíqueme sus 6 patrones por categoría GoF."
Creacionales: Singleton y Factory Method. Estructurales: Composite y Bridge. De comportamiento: Template Method y Observer. Dos de cada categoría — saber la clasificación de memoria suma puntos fáciles.
"Su Bridge me parece Strategy. Convénzame."
1) Comparten estructura; los distingue la intención. 2) Aquí hay dos jerarquías que evolucionan solas: pedidos (Mostrador/Delivery con cálculos de precio propios) y pagos (Tarjeta/Yape/Efectivo). 3) El objetivo es estructural: evitar n×m clases combinadas (2+3 en vez de 2×3). 4) Si solo hubiera UNA clase Pedido cambiando de algoritmo de pago, ahí sí sería Strategy. Cierre: "GoF mismo señala que varios patrones difieren solo en el propósito."
"¿Qué patrón elimina el if que ustedes NO escribieron?" (pregunta abstracta)
Tres patrones matan tres ifs distintos: Factory Method mata el switch(tipoProducto); Bridge mata el switch(tipoPago); Template Method mata los if (tipo == helado) no cocinar. Bonus: el pago por defecto PagoNoSeleccionado mata el if (metodoPago != null). El dominio quedó con cero condicionales de tipo — verificable con una búsqueda.
"Deme el recorrido de UNA llamada: sistema.coordinarPreparacion(pedido). ¿Qué patrones se activan y en qué orden?"
① La UI llama a SistemaHamburgueseria, que delega en pedido.preparar() → Composite. ② El pedido hace cambiarEstado(EN_PREPARACION) → Observer notifica a Cocina/Despacho/Pantalla. ③ El for llama alimento.preparar() de cada hijo → Template Method ejecuta sus 5 pasos. ④ comprobarIngredientes() descuenta del inventario compartido → que existe único gracias al Singleton. ⑤ Al terminar, cambiarEstado(LISTO) → Observer otra vez. Cuatro patrones en una sola llamada.
"¿Qué pasa si no hay stock a mitad de un pedido?"
Inventario.descontarIngredientes() valida TODO antes de descontar (primero stream de verificación, luego descuento) para no dejar descuentos a medias. Si falta algo lanza StockInsuficienteException; Pedido.preparar() la captura, cambia el estado a SIN_STOCK y el Observer avisa a todas las áreas. La excepción es checked-style propia del dominio.
"Critique su propio código: ¿qué mejorarían?" (la trampa de humildad)
Tener 3-4 respuestas honestas preparadas desarma la pregunta:
① preparar() del Template debería ser final para blindar el orden.
② Al Observer le falta desuscribir() (attach sin detach).
③ Pedido.getPrecio() usa un monto acumulado; el Composite puro sumaría los hijos en el momento.
④ El modelado simplifica ingredientes (el Refresco descuenta Coca Cola E Inca Kola a la vez).
Cerrar con: "Ninguna afecta el funcionamiento actual y las tenemos identificadas como mejoras."
"¿Por qué Pedido EXTIENDE PedidoAbstracto pero IMPLEMENTA Alimento?"
Porque juega en dos patrones a la vez: hereda de PedidoAbstracto para ser la abstracción del Bridge (recibe el campo metodoPago y el contrato procesarPago), e implementa Alimento para ser el composite del Composite (tratarse igual que un producto). Java no permite herencia múltiple de clases: la línea class Pedido extends PedidoAbstracto implements Alimento es literalmente el punto donde dos patrones se cruzan.
"Muéstreme el patrón X en los ejercicios de clase, no en el proyecto."
· Factory Method: patrones/FactoryFigura + FactoryCirculo/Cuadrado/Triangulo/Compuesto.
· Composite: patrones/Compuesto — pintar() recorre la lista de Figuras.

"¿Singleton no es considerado un antipatrón hoy en día?"
Reconocer la crítica y acotarla: "Se le critica cuando se abusa de él como variable global o cuando esconde dependencias en cientos de clases. Aquí su alcance es mínimo — un solo punto de acceso al inventario compartido, usado desde una sola clase coordinadora — y el proyecto es una app de escritorio sin inyección de dependencias, donde es la solución proporcional al problema."
